# FSharpLint Delegation Criteria

As established in Phase 0 of the `fs-assay` plan, FSharpLint is not automatically trusted for rule verification. Because `fs-assay` acts as a strict, determinism-focused gatekeeper for AI-generated code, delegating rules to FSharpLint is conditional upon strict determinism and reliability gates.

## Delegation Gates

A rule can only be delegated to FSharpLint if it meets **all** of the following criteria:

1. **Deterministic Execution**: The rule must produce the exact same outcome on the identical AST/TAST across different environments and platforms. Heuristic-based guessing is forbidden.
2. **Reliability and Fallback Status**: The rule must not silently swallow exceptions. If FSharpLint fails to load or crashes during the execution of a delegated rule, it must result in a `ToolFailure` or `Inconclusive` verdict, never a silent `Pass`.
3. **Corpus-Verified**: The rule must be verified against a known, version-pinned test corpus ensuring no false positives on acceptable idiomatic F# constructs.
4. **No Side-Effects on AST**: The rule must be strictly analytical and pure. Fixes generated by the rule must be deterministic and reversible.

## Inventory and Implementation Status

For any rule that fails these gates (e.g., due to version incompatibility, exception swallowing, or heuristic false positives), the rule will be implemented as a native `FsAssay.Analyzers` AST/TAST inspection rule rather than delegated.

*Current Status*: FSharpLint delegation remains corpus-gated and conditional.
