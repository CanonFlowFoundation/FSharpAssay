# FsAssay: Agent Context & Project Essence

## 🎯 Core Identity & Purpose
FsAssay is a deterministic verification tool acting as an automated "Senior F# Architect". Its primary purpose is to ensure that F# code—especially code generated by AI coding agents—strictly adheres to idiomatic, elite functional architecture. It actively prevents agents from falling back into C#-style OOP patterns (such as mutability, `null` references, partial functions like `Option.get`, and unguarded exceptions).

Unlike generic linters or formatters, FsAssay is:
- **Strictly Mechanical**: Backed by `FSharp.Compiler.Service` (FCS) and `FSharp.Analyzers.SDK`, it analyzes Typed Abstract Syntax Trees (TAST) and ASTs for undeniable proof of code shapes and semantics.
- **Architecturally Opinionated**: It enforces different strictness levels based on profiles (`core` must be pure, `shell` allows ORM/effect adapters, `interop` allows boundary pragmatism).
- **Uncompromising**: FsAssay has a 4-state verdict (`Pass`, `Fail`, `Inconclusive`, `ToolFailure`). A missing tree or crashed analyzer results in `ToolFailure`, never a silent `Pass`.

## 🏗 Source Code Architecture (`src`)

The repository is modularized into specialized components:

### 1. `FsAssay.Runner` (The Orchestrator)
The heart of the application, responsible for project evaluation and lifecycle:
- **`Program.fs`**: CLI entry point for full verification passes (`verify`) or provisional checks (`check`).
- **`Domain.fs`**: Defines the rigorous verdict model (`AssayVerdict`, `RuleEvaluation`, `Finding`), ensuring clear agent/judge separation.
- **`Orchestrator.fs`**: Coordinates the compiler, projects, and rules.
- **`AutoFix.fs`**: Drives deterministic, semantics-preserving automated code fixes (`--fix`).
- **`ProjectSystem.fs`**: Interfaces with MSBuild to load F# projects and their required target frameworks (TFMs).
- **`Output.fs` & `Server.fs`**: Exports findings into Markdown Rate Cards, Material Design 5 HTML dashboards, and standardized OASIS SARIF v2.1.0 evidence bundles.

### 2. `FsAssay.Analyzers` (The Rule Engine)
- Contains the custom F# rule implementations built on top of `FSharp.Analyzers.SDK`.
- Implements hybrid context indexing: scanning AST for scope (loops, async expressions) and TAST for semantic symbol resolution (e.g., distinguishing a domain `get` from `Option.get`).

### 3. `InspectTAST` (Developer Tooling)
- A utility for dumping and inspecting the F# compiler's Typed Abstract Syntax Tree, used to design new rules and understand compiler output shapes.

### 4. `Specimens` (The Test Corpus)
- Contains extensive categorized examples of both bad anti-patterns and good idiomatic F# code (e.g., `CsharpishOrderProcessor.fs` vs `StylishOrderProcessor.fs`).
- Acts as the baseline against which FsAssay rules are proven.

### 5. `FsAssay.Tests`
- Expecto-based zero-regression test suite validating the 42+ active rules.

## 📖 Key Directives from Documentation (`md`)

### The 6 Laws of FsAssay (`FSharpAssay.md`)
1. **Agent/Judge Separation**: An AI cannot modify the rules, severities, or CI gates in the same change it is trying to pass.
2. **Honest Uncertainty**: Missing evidence or tool failures result in `Inconclusive` or `ToolFailure`, never `Pass`.
3. **Deterministic Blocking**: Only deterministic rules block builds. Contextual rules block only with explicit policy.
4. **Reproducible Verdicts**: Same source + same policy + same toolchain = same verdict.
5. **Visible Suppression**: All ignores and suppressions must be recorded in the SARIF/JSON evidence bundle. No silent skips.
6. **Full Release Authority**: Provisional checks do not grant release authority. Only a full `--verify` across all TFMs counts.

### Anti-Pattern Rule Categories (`README.md`)
- **Tier 1: SOTA Correctness**: Blocks `Unchecked.defaultof`, partial access (`Option.get`), async blocking, and unguarded exceptions.
- **Tier 2: SOTA Security**: Blocks hard-coded secrets, swallowed exceptions, and path traversals.
- **Core Functional Smells**: Blocks primitive obsession (use DUs instead), `mutable`, generic exception catching, imperative `while` loops, and OOP inheritance.

## 🤖 Workflow Rule for Agents
When an agent writes F# code in this ecosystem:
1. Always write **idiomatic, expression-oriented** F# (Records, DUs, `Result`, `Option`, `async`).
2. Run `dotnet run --project FsAssay.Runner -- /path/to/code` after editing to obtain your grade.
3. Fix all issues surfaced by the examiner before seeking human review. Never submit C#-shaped F# code.
