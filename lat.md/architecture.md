# architecture
This document captures the essence, architecture, and core laws of FsAssay. It serves as the primary context for agents working on this project.

## Core Identity
FsAssay is a deterministic verification tool acting as an automated "Senior F# Architect".

It enforces idiomatic, elite functional architecture on F# code generated by agents, preventing C#-style OOP patterns like mutability, `null`, and unguarded exceptions. It uses the `FSharp.Compiler.Service` and `FSharp.Analyzers.SDK` for strict validation.

## Source Code
The codebase is modularized into specialized components: Runner, Analyzers, InspectTAST, Specimens, and Tests.

### Runner
The heart of the application, responsible for project evaluation and lifecycle.

It coordinates the compiler, loads MSBuild projects via `ProjectSystem.fs`, orchestrates verification (`Orchestrator.fs`), handles fixes (`AutoFix.fs`), and outputs SARIF and HTML dashboards (`Output.fs`, `Server.fs`).

### Analyzers
The custom F# rule engine built on top of `FSharp.Analyzers.SDK`.

It implements hybrid context indexing by scanning both the AST for scope and the TAST for semantic symbol resolution.

### Developer Tools
Utilities for developing and inspecting FsAssay rules.

The `InspectTAST` tool dumps the F# compiler's Typed Abstract Syntax Tree to help developers understand compiler output shapes.

### Test Corpus
Extensive examples of anti-patterns and elite F# idioms.

The `Specimens` directory acts as the baseline against which FsAssay rules are proven.

### Tests
Expecto-based zero-regression test suite validating the active rules.

## Directives
Key principles and laws governing the FsAssay architecture.

### The 6 Laws
The six immutable laws of FsAssay from `FSharpAssay.md`.

1. **Agent/Judge Separation**: AI cannot modify rules while trying to pass.
2. **Honest Uncertainty**: Missing evidence is `Inconclusive`.
3. **Deterministic Blocking**: Only deterministic rules block.
4. **Reproducible Verdicts**: Same inputs = same verdict.
5. **Visible Suppression**: All ignores must be recorded.
6. **Full Release Authority**: Provisional checks don't grant release.

### Anti Pattern Rules
Categories of rules enforced by FsAssay.

Tier 1 blocks `Unchecked.defaultof`, partial access, and unguarded exceptions. Tier 2 blocks hard-coded secrets and swallowed exceptions. Core Functional Smells block primitive obsession, `mutable`, and imperative loops.

### Machine Learning & Profile Boundaries
Strict architectural boundaries mapped directly to ML functionality.

`FSA-ML01` and `FSA-ML02` force ML code to use pure Tensors and DUs instead of raw arrays and OOP inheritance. `FSA-B01` strictly confines mutability (like `Dictionary` or `System.Array`) to the `[<Profile("shell")>]` boundary, preventing effect leakage into the pure functional core.

## Agent Workflow
Guidelines for agents contributing to FsAssay or target F# codebases.

Agents must write idiomatic, expression-oriented F# (Records, DUs). After editing, run `dotnet run --project FsAssay.Runner` to get a grade and fix issues before review.
